= Автоматическое обновление =\
\
== Использование автоматического обновления ==\
Посетите эту станицу: {{YacyLink\|ConfigUpdate\_p.html}}\
\
You can either manually choose a release from the updatelocations and
update or enable the autoupdate feature. The autoupdate regularly
(configure the interval) checks the updatelocations. You can choose
between main (stable snapshot-releases \~ every 2 months) or the current
dev releases. The dev releases ending on 123 are often more experimental
than the others, so they are blacklisted by default.\
\
Вы можете либо вручную выбрать освобождение от updatelocations и
обновить или включить автообновление. В автообновление регулярно
(настроить интервал) проверяет updatelocations. Вы можете выбрать между
основным (стабильным снимок-релизы \~ каждые 2 месяца) или текущего
разрабатываемых версий. В dev-релизы, оканчивающиеся на 123 часто
являются более экспериментальные, чем остальных, поэтому они находятся в
черном списке по умолчанию.\
\
== Настроить местоположение своих обновлений ==\
\
An updatelocation is just a HTML-Page with links to yacy-tarballs
conforming to the version-scheme. The updatelocation is configured per
network in the network-definition (for example
defaults/yacy.network.freeworld.unit):\
\
В updatelocation-это просто HTML-страницы со ссылками на yacy-TAR-файлов
соответствующей версии-схема. В updatelocation настроен в сети в
сети-определения (например по умолчанию/yacy.сети.freeworld.блок):\
\
\<pre\>\
network.unit.update.location0 = http://your.domain.tld/yacyreleases/\
\</pre\>\
\
==Setup signatures==\
To be sure, YaCy really downloads the tarballs, the updatelocation admin
uploaded, you should provide signatures for your releases. DNS-spoofing,
man-in-in-the-middle-attacks or attacks on the webserver are not that
difficult.\
\
==Настройка подписей==\
Чтобы быть уверенным, YaCy действительно загрузок TAR-архивы, в
updatelocation админ загружен, вы должны предоставить подписи для ваших
релизов. Служба DNS-спуфинг, мужчина-в-в-середине-или атаки на сервер не
так сложно.\
\
=== Создание частного и публичного ключа ==\
\
Настроить место, куда поставить закрытый ключ с имуществом (Configure
the location where to put the private key with the property)
\'\'privateKeyFile\'\' in \'\'build.properties\'\' . Затем выполните:\
\
\<pre\>ant genkey\</pre\>\
\
Частный ключ будет создан в указанном месте. Публичный ключ будет иметь
расширение \'\'.pub\'\'.\
\
{{Note\|По умолчанию значение для privateKeyFile будет \"private.key\" и
эти приватный и публичный ключи будут созданы в папке yacy}}\
\
=== Помещение публичного ключа в networkdefinition ===\
\
The network definition (Сети определение) файла должно выглядеть
следующим образом:\
\
\<pre\>\
network.unit.update.location0 = http://your.domain.tld/yacyreleases/\
network.unit.update.location0.key = MIIBuDCCASwGQeEwx7V\...(very
long)\...bukaPtQxr2p9y1QNZFauihmu4ak4AyT\
\</pre\>\
\
Now, YaCy will try to download the signature and check it. So provide
\'\'.sig\'\'-files!\
\
{{Note\|The default network definitions file is located in
defaults/yacy.network.freeworld.unit.\
But the values can be changed during runtime as well here:
http://localhost:8090/ConfigProperties\_p.html}}\
\
=== (Генерировать распределение TAR-архива с подписью) Generate a
distribution tarball with signature===\
\
Выполните:\
\
\<pre\>ant clean dist sign\</pre\>\
\
B Вы найдете архив и .sig-файл в релизе-каталог. Положить их в
updatelocation\
\
You\'ll find the tarball and a .sig-file in the RELEASE-directory. Put
them on the updatelocation
